# Test Agent Desktop AI Agent 测试策略 V1

## 1. 文档目的

本文档从测试 Leader 视角定义 Test Agent Desktop V1 的测试策略和测试方法。测试重点是 AI Agent 能力是否正确、可信、稳定、可控和可追溯；Desktop 页面仅作为用户操作入口进行必要验证，不作为 V1 的主要测试投入方向。

V1 需要回答以下核心问题：

1. Agent 是否正确理解需求文档和用户补充信息。
2. Agent 是否能生成完整、准确、可执行的测试点和测试用例。
3. RAG 是否召回正确知识，并被 Agent 正确使用。
4. Agent 工作流是否按照预期完成检索、生成、人工确认和再次生成。
5. 本地模型与云模型切换后，能力和数据边界是否正确。
6. 模型异常、知识冲突、输出格式错误时，系统是否可控并能给出明确结果。
7. 每次生成使用了什么输入、知识和执行节点，是否可以追溯。

## 2. 项目 AI Agent 能力范围

当前项目的核心链路为：

```text
需求文档
→ Markdown 标准化
→ 补充需求合并
→ RAG 检索
→ 测试点生成
→ 人工确认或编辑
→ 基于确认后的测试点再次检索
→ 测试用例生成
→ 结果持久化与导出
```

V1 纳入以下 AI Agent 能力：

- 需求上下文构建。
- 规则生成与模型生成。
- Ollama 本地模型和 OpenAI-compatible API 模型路由。
- 测试点生成。
- 测试用例生成。
- 测试点人工确认后的状态传递。
- 关键词 RAG 和本地向量 RAG。
- Prompt 模板和结构化 JSON 输出。
- Agent Workflow 状态编排。
- RAG 轨迹和运行结果持久化。
- Skill 加载、调用和结构化错误返回。

V1 不以自动执行测试、多人协作、企业权限和插件市场为测试目标。

## 3. 测试目标

### 3.1 正确性

- 原始需求、补充需求和 RAG 上下文被完整传入正确生成阶段。
- 测试点和测试用例符合约定的数据结构。
- 人工编辑后的测试点是测试用例生成的唯一有效上游数据。
- 模型来源、模型名称和请求协议与当前配置一致。
- 导出和持久化结果与最终确认结果一致。

### 3.2 业务质量

- 测试点覆盖需求中的主流程、反向、边界、异常、数据、权限和兼容性风险。
- 测试用例具备明确前置条件、可执行步骤和可验证预期结果。
- 生成内容不虚构需求中不存在的业务规则。
- 测试点、测试用例和原始需求之间能够建立追溯关系。

### 3.3 稳定性

- 相同输入多次生成时，关键需求覆盖保持稳定。
- 模型超时、网络失败、非法 JSON 和字段缺失不会产生伪成功结果。
- RAG 索引为空、召回为空或 Embedding 服务不可用时，系统行为符合所选模式。
- 长文档和较大知识库下，工作流可以完成且不会污染上一次运行状态。

### 3.4 可控性与可观测性

- 用户可以明确选择是否启用模型生成和 RAG。
- Ollama 与 OpenAI-compatible API 两种模式互斥。
- 测试点必须允许人工确认或编辑后再生成用例。
- 工作流执行节点、RAG Query、召回内容来源和生成结果可以记录。
- 错误信息能够定位到模型、RAG、Skill、格式解析或文件处理阶段。

### 3.5 安全性

- API Key 不出现在日志、异常、Prompt、运行结果和导出文件中。
- 外部知识中的指令不能改变系统预期任务或诱导泄露敏感信息。
- Skill 执行失败不能影响主进程和其他 Skill。
- 文件名、运行编号和 Skill 名称不能造成越权文件访问或覆盖。

## 4. 风险分级

| 等级 | 风险 | 影响 |
|---|---|---|
| P0 | Agent 遗漏关键业务规则或生成错误预期结果 | 直接导致测试设计失真 |
| P0 | 人工修改测试点后仍使用旧数据生成用例 | 用户确认机制失效 |
| P0 | 模型返回非法结构但系统显示成功 | 产生不可用或错误结果 |
| P0 | API Key 或敏感需求内容泄露 | 数据安全事故 |
| P1 | RAG 召回错误知识或知识来源不可追溯 | Agent 生成内容被错误上下文污染 |
| P1 | 本地模型和云模型路由错误 | 请求发送到非预期服务 |
| P1 | 多次运行状态串扰 | 当前结果混入历史任务数据 |
| P1 | 模型超时、断网或 Ollama 不可用时错误不明确 | 用户无法判断失败阶段 |
| P2 | 同一输入生成结果波动过大 | 回归结果不稳定 |
| P2 | 大文档或较大索引导致响应时间不可接受 | 本地使用体验下降 |
| P3 | 页面布局或非关键交互问题 | 不影响 Agent 核心能力 |

测试投入遵循 P0 全覆盖、P1 重点覆盖、P2 基线监控、P3 必要冒烟的原则。

## 5. AI Agent 测试分层

### 5.1 L1：组件级确定性测试

目标是验证不依赖真实模型的确定性逻辑，使用 Fake Router、Fake HTTP Client、Fake Embedding Adapter 和固定响应。

重点范围：

- Prompt 是否包含需求、补充需求、RAG 上下文和模板字段。
- 模型请求的 URL、Header、Model、Message 和参数是否正确。
- Ollama 与 API Provider 路由是否互斥。
- 模型 JSON 到 TestPoint、TestCase 实体的字段映射。
- 非法 JSON、缺字段、错误字段类型和空结果处理。
- RAG 分块、Top-K、相似度阈值、文档删除和向量删除。
- Workflow 节点顺序和状态变化。
- 人工编辑测试点后旧用例失效。
- Result Store 中测试点、用例和 RAG 轨迹的一致性。
- Skill 加载、调用和结构化错误结果。

该层作为每次提交必须执行的快速回归集。

### 5.2 L2：Agent 服务集成测试

以 ApplicationService 为入口，验证多个组件之间的真实数据传递，不经过 Desktop 页面。

核心场景：

1. 文档转换后，需求正文和补充需求共同进入测试点生成。
2. 开启 RAG 后，第一次召回用于测试点生成。
3. 用户编辑测试点后，系统清空旧用例。
4. 确认后的测试点触发第二次 RAG 召回。
5. 第二次召回和确认后的测试点共同进入用例生成。
6. 最终结果、执行节点和两阶段 RAG 轨迹被持久化。
7. Markdown、Excel 导出与当前内存中的最终用例一致。
8. 连续执行两个不同项目时，上下文和结果不串扰。

该层重点验证 Agent 工作流，而不是单个函数。

### 5.3 L3：受控真实模型测试

分别接入一个固定 Ollama 模型和一个固定 OpenAI-compatible API 模型，使用固定数据集执行受控验证。

测试内容：

- 模型连接和模型发现。
- Prompt 能否被真实模型正确理解。
- 结构化 JSON 成功率。
- 必填字段完整率。
- 测试点和测试用例的基础业务质量。
- 中文需求、长需求和混合中英文需求表现。
- 首次冷启动和后续热调用耗时。
- 超时、断网、错误模型名和服务不可用。
- 本地模型与云模型结果差异。

真实模型测试不要求文本逐字一致，应使用结构硬断言和质量指标评价。

### 5.4 L4：AI Agent 端到端验收

使用固定需求和固定知识库完成完整业务闭环：

```text
导入需求
→ 导入知识库
→ 输入补充规则
→ 开启 RAG
→ 生成测试点
→ 人工修改一个测试点
→ 生成测试用例
→ 保存运行结果
→ 导出 Markdown 和 Excel
```

端到端验收应覆盖以下组合：

| 生成方式 | RAG 模式 | 验证目标 |
|---|---|---|
| 规则生成 | 关闭 | 基础兜底链路 |
| 规则生成 | 关键词检索 | 确定性 RAG 链路 |
| Ollama | 关闭 | 本地模型基础能力 |
| Ollama | 向量检索 | 本地完整 Agent 链路 |
| API 模型 | 关闭 | 云模型基础能力 |
| API 模型 | 向量检索 | 云模型完整 Agent 链路 |

### 5.5 L5：页面必要验收

页面测试只验证 Agent 能力能否被正确触发和展示：

- 模型来源切换后配置项互斥。
- RAG、模型生成开关真实影响后台执行。
- 生成期间重复操作被限制，页面不假死。
- 测试点编辑结果能够同步到服务层。
- 错误信息、RAG 轨迹和生成结果正确展示。
- 导出的内容与页面最终确认内容一致。

非关键样式、布局细节和一般视觉兼容性不作为 V1 重点。

## 6. AI 输出质量评估方法

### 6.1 结构硬断言

以下指标必须达到 100%：

- 输出能够解析为约定 JSON。
- 必填字段存在且类型正确。
- 用例编号在单次结果中唯一。
- 每条用例包含测试步骤和预期结果。
- 模块、功能点、优先级和用例类型符合约束。
- 测试点编辑内容出现在下游输入中。
- RAG 召回结果包含可识别来源。
- 失败结果不会保存为成功运行。

### 6.2 内容质量指标

| 指标 | 计算方式 | V1 建议门槛 |
|---|---|---|
| 关键需求覆盖率 | 已覆盖关键需求数 / 关键需求总数 | ≥ 95% |
| 一般需求覆盖率 | 已覆盖一般需求数 / 一般需求总数 | ≥ 85% |
| 高风险场景覆盖率 | 已覆盖高风险场景数 / 标注高风险场景总数 | ≥ 90% |
| 幻觉率 | 无输入依据的业务断言数 / 业务断言总数 | ≤ 5% |
| 用例可执行率 | 可直接执行用例数 / 用例总数 | ≥ 90% |
| 重复率 | 重复或近似重复用例数 / 用例总数 | ≤ 10% |
| 需求追溯率 | 可关联到需求依据的用例数 / 用例总数 | ≥ 90% |
| JSON 成功率 | 成功解析次数 / 总调用次数 | ≥ 99% |

门槛用于 V1 基线，后续根据固定评估集的实测结果调整。

### 6.3 评分方式

AI 输出采用三类评价结合：

1. 程序自动评价：结构、字段、编号、枚举值、来源和重复项。
2. 规则评价：关键词覆盖、风险类型覆盖和需求追溯。
3. 测试专家评价：业务正确性、可执行性、预期结果准确性和幻觉。

测试专家评价使用统一评分表，避免只凭“看起来不错”判断模型质量。

## 7. RAG 专项测试

### 7.1 检索正确性

- Query 同时包含当前需求和补充需求。
- 测试点阶段与用例阶段使用各自正确的 Query。
- Top-K 数量正确。
- 相似度低于阈值的内容不返回。
- 删除知识文档后，关键词索引和向量索引均不再召回该文档。
- 每条召回内容保留来源文件。

### 7.2 检索质量

建立带标准答案的 Query 集，评价：

- Recall@K：标准相关知识是否进入前 K 条。
- Precision@K：前 K 条中相关知识的比例。
- 来源正确率：召回内容与标记来源是否一致。
- 空召回正确率：不存在相关知识时是否正确返回空结果。

### 7.3 生成利用效果

RAG 测试不能只检查“召回到了”，还需要比较：

- 关闭 RAG 时的生成结果。
- 开启 RAG 时的生成结果。
- 开启 RAG 但召回无关内容时的生成结果。

重点确认正确知识提高了需求覆盖，错误知识不会被无条件采纳。

### 7.4 知识冲突与 Prompt 注入

- 知识库内容与当前需求冲突时，以当前需求和用户补充说明为准。
- 两份知识文档互相冲突时，结果需保留来源并避免给出无依据结论。
- 知识文档中包含“忽略原任务”“输出密钥”等指令时，Agent 不得执行。
- 恶意内容不能改变输出结构或模型路由。

## 8. Agent 工作流专项测试

### 8.1 节点与状态

验证以下状态转换：

```text
初始上下文
→ retrieve_context
→ generate_test_points
→ 用户确认或编辑
→ retrieve_context_for_cases
→ generate_test_cases
→ persist_result
```

需要检查：

- RAG 关闭时不执行检索节点。
- 模型生成关闭时走规则生成。
- 模型生成开启时不提前生成规则用例。
- 编辑测试点后旧用例被清空。
- 任一节点失败后，后续依赖节点不执行。
- 新任务开始时不复用上一任务的测试点、用例和 RAG 轨迹。

### 8.2 中断与恢复

- 模型请求超时后能够明确结束当前任务。
- 用户关闭窗口或任务对象释放时，不产生伪成功信号。
- 重试不会重复写入同一运行结果。
- 异常结束后可以重新发起新任务。

### 8.3 LangGraph 与默认工作流一致性

如果启用 LangGraph，可选实现必须与默认 AgentWorkflow 保持以下业务契约一致：

- 输入上下文一致。
- RAG 开关语义一致。
- 测试点和测试用例结构一致。
- 节点失败语义一致。
- 最终持久化内容一致。

## 9. 模型路由与故障测试

### 9.1 路由正确性

- Ollama 模式只使用本地模型配置。
- API 模式只使用 Base URL、API Key 和 API Model。
- Embedding 模型不能用于对话生成。
- 对话模型不能误用为向量模型。
- 切换模型来源后不残留上一模式的有效凭据。

### 9.2 故障场景

| 故障 | 预期结果 |
|---|---|
| Ollama 未启动 | 明确提示连接失败，不生成空结果 |
| 本地模型不存在 | 指出模型不可用 |
| API Base URL 错误 | 返回网络或 HTTP 错误 |
| API Key 无效 | 返回鉴权失败且不泄露 Key |
| 模型请求超时 | 当前任务失败，不自动产生重复调用 |
| 模型返回空字符串 | 判定生成失败 |
| 模型返回 Markdown 代码块包裹 JSON | 按约定解析或明确失败 |
| 模型返回字段缺失 | 指出结构错误 |
| 模型返回超长内容 | 不阻塞主线程，结果可控 |

## 10. 测试数据策略

建立版本化的固定评估集，每条数据包含：

- 原始需求文档。
- 用户补充需求。
- 可选知识库文档。
- 关键需求清单。
- 高风险场景清单。
- 标准测试点。
- 最低可接受用例集合。
- 禁止出现的幻觉内容。
- 预期召回文档和片段。

V1 数据集至少覆盖：

1. 普通 Web 功能需求。
2. 具有角色和权限的需求。
3. 状态流转需求。
4. 金额、时间、数量等边界需求。
5. 接口和异步回调需求。
6. 包含冲突说明的需求。
7. 信息不完整的需求。
8. 长文档和多模块需求。
9. 中文与英文混合需求。
10. 包含历史缺陷知识的需求。

评估集不得直接用于修改 Prompt 后的人工挑选结果，避免只对少量样本过拟合。

## 11. 非功能测试

### 11.1 性能

记录以下基线：

- 文档转换耗时。
- 知识库分块和向量化耗时。
- 单次 RAG 检索耗时。
- 测试点生成耗时。
- 测试用例生成耗时。
- Ollama 冷启动与热调用耗时。
- 连续运行后的内存变化。

### 11.2 稳定性

- 同一固定输入连续生成 20 次，统计成功率和质量波动。
- 连续处理多个不同项目，检查状态隔离。
- 大文档与大知识库组合运行。
- 模型超时后立即执行下一任务。

### 11.3 安全

- API Key 存储、日志、异常和导出泄露检查。
- Prompt 注入和知识库注入。
- Skill 非法入口、异常返回和越权文件访问。
- 文件路径穿越和运行编号路径注入。
- 运行结果中的敏感需求内容边界。

## 12. 回归测试策略

### 12.1 每次提交

- 全部组件级确定性测试。
- ApplicationService 核心链路测试。
- Prompt 和 JSON Schema 契约测试。
- RAG 基础检索与删除测试。
- 配置和密钥脱敏测试。

### 12.2 Agent、Prompt 或模型适配改动

除基础回归外，执行：

- 固定评估集质量回归。
- 本地模型受控测试。
- API 模型受控测试。
- JSON 成功率和质量指标对比。
- 关闭 RAG 与开启 RAG 的对照测试。

### 12.3 RAG 或 Embedding 改动

除基础回归外，执行：

- Recall@K、Precision@K 对比。
- 阈值和 Top-K 边界。
- 索引新增、覆盖、删除和重建。
- 两阶段 RAG Query 和轨迹验证。
- 无关知识、冲突知识和恶意知识测试。

### 12.4 发布前

- 完整 Agent 端到端矩阵。
- 固定评估集质量验收。
- 真实 Ollama 与 API 冒烟。
- Windows/macOS 目标环境冒烟。
- 安全和敏感信息检查。
- Markdown、Excel 最终结果一致性检查。

## 13. V1 质量准入与退出标准

### 13.1 测试准入

- 需求和 Agent 能力范围已经确认。
- 模型、Prompt 和输出结构版本固定。
- 固定评估集已经评审。
- Ollama 和 API 测试模型明确。
- 测试环境依赖安装完整。

### 13.2 发布退出

- P0 测试 100% 通过，不存在未关闭 P0/P1 缺陷。
- 组件级和服务集成自动化测试全部通过。
- Agent 完整链路六种组合至少完成一次验收。
- 结构硬断言 100% 通过。
- 内容质量指标达到 V1 门槛。
- RAG 来源可追溯，两阶段轨迹完整。
- API Key 和敏感信息检查通过。

## 15. V1 总结

本项目的测试重点不是验证页面按钮是否可以点击，而是验证 Agent 能否在受控工作流中稳定地产出可信测试设计。

V1 采用以下核心策略：

```text
确定性组件测试保证代码逻辑
→ 服务集成测试保证 Agent 数据流和状态流
→ 固定评估集衡量生成质量
→ RAG 专项验证知识召回和利用
→ 真实模型测试验证实际能力与故障行为
→ 少量端到端和页面验收保证用户闭环
```

最终以“正确、可信、稳定、可控、可追溯、安全”作为 AI Agent 能力的统一质量标准。
